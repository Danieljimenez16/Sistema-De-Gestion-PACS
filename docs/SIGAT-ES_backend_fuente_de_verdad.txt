SIGAT-ES - Fuente de verdad funcional y técnica para Backend
Documento de alineación Frontend -> Backend
Versión inicial basada en el preliminar del proyecto

Objetivo de este documento
Este documento consolida, interpreta y aterriza técnicamente el preliminar del proyecto SIGAT-ES para que el backend pueda diseñar y exponer una API consistente, escalable y utilizable por frontend sin retrabajo innecesario. Su propósito es convertir el texto del preliminar en una fuente de verdad funcional y técnica para definir módulos, entidades, campos, endpoints, reglas de negocio, validaciones, filtros, contratos de respuesta y consideraciones de arquitectura.

Alcance de esta fuente de verdad
Este documento sí define:
- Qué problema resuelve el sistema.
- Qué módulos mínimos debe exponer el backend.
- Qué entidades principales existen.
- Qué campos deberían manejarse por módulo.
- Qué endpoints necesita frontend para operar sin ambigüedad.
- Qué validaciones y restricciones deben considerarse desde backend.
- Qué relaciones de datos deben modelarse para evitar romper la base de datos.
- Qué contratos generales de respuesta conviene estandarizar.
- Qué decisiones conviene tomar para facilitar QA, auditoría, reportes y escalabilidad.

Este documento no define todavía:
- Integraciones externas complejas.
- Automatización avanzada de inventario por red.
- Módulos financieros complejos.
- Diseño visual del frontend.

1. Comprensión profunda del problema del negocio
Element System necesita dejar atrás el control manual y disperso de activos tecnológicos, que hoy vive en hojas de cálculo y registros por área. El preliminar menciona explícitamente problemas de trazabilidad, duplicidad de información, errores de actualización y dificultad para conocer el estado real de los recursos. También menciona que actualmente se controlan routers, antenas, servidores, dispositivos, laptops y licencias. Por lo tanto, el sistema no debe pensarse como un simple CRUD de inventario, sino como una plataforma de control operativo de activos tecnológicos con trazabilidad, responsables, ubicaciones, estados, licencias, reportes y auditoría.

La pregunta de negocio que el sistema debe responder rápido es:
- Qué activos tenemos.
- Dónde están.
- En qué estado están.
- Quién los tiene o quién es responsable.
- Qué licencias están asociadas.
- Qué cambios se hicieron y cuándo.

Eso convierte al backend en el eje de consistencia del sistema. No puede limitarse a guardar registros planos. Debe modelar relaciones y reglas de integridad que permitan evitar duplicados, inconsistencias y pérdida de trazabilidad.

2. Objetivo real del sistema a nivel backend
El backend debe permitir como mínimo:
- Registrar activos tecnológicos.
- Consultarlos con filtros.
- Editarlos sin perder historial.
- Cambiar su estado.
- Asignarlos a responsable, área y ubicación.
- Gestionar licencias asociadas a activos o usuarios.
- Registrar auditoría de cambios relevantes.
- Proveer reportes básicos.
- Soportar roles y autenticación.
- Soportar migración inicial desde inventario base.

3. Principios técnicos que backend debe asumir desde el inicio
3.1 Arquitectura
El preliminar menciona MVC como estándar de arquitectura. En Node.js esto se puede aterrizar así:
- routes/: definición de endpoints y mapping HTTP.
- controllers/: validación de request y coordinación.
- services/: lógica de negocio real, para no ensuciar controllers.
- repositories o models/: acceso a base de datos.
- middlewares/: auth, roles, manejo de errores, validaciones.
- validators/: reglas reutilizables de payload y query params.

Aunque el preliminar menciona MVC, la mejor práctica en Node es no dejar toda la lógica en controller. Se recomienda una variante de MVC limpia con capa de servicio.

3.2 Base de datos
Debe ser relacional, porque el preliminar lo indica y porque el dominio requiere integridad entre activos, tipos, ubicaciones, responsables, estados, licencias, asignaciones y auditoría. Lo más sano es modelar tablas normalizadas y solo desnormalizar donde ayude a reportes.

3.3 Trazabilidad
No se debe sobrescribir información crítica sin dejar historial. Cambios de estado, reasignaciones, modificaciones sensibles y relaciones con licencias deben dejar registro.

3.4 Clean code y mantenibilidad
- Nombres consistentes y semánticos.
- Controladores delgados.
- Validación fuerte en backend.
- Respuestas API uniformes.
- Soft delete donde aplique.
- Catálogos controlados para estados, tipos y roles.
- Paginación y filtros desde el primer día.

4. Módulos funcionales que sí se desprenden del preliminar
A partir del preliminar, los módulos mínimos del MVP son:

4.1 Autenticación y autorización
Porque el documento exige acceso por roles y autenticación.

4.2 Inventario de activos
Porque el núcleo del sistema es registro, consulta, actualización y clasificación de activos.

4.3 Asignación y estados
Porque el documento menciona responsable, área, ubicación, historial de asignaciones y bitácora de cambios de estado.

4.4 Licencias
Porque se requiere control de licencias, vigencia, asignación y alertas simples.

4.5 Auditoría
Porque el documento exige historial de cambios y consultas por usuario/fecha/acción.

4.6 Reportes
Porque se solicitan reportes esenciales y exportación.

4.7 Migración inicial
Porque se menciona importación inicial desde hojas de cálculo y limpieza/normalización del inventario.

4.8 Catálogos base
No aparece como módulo de cara al usuario, pero backend lo necesita para no hardcodear opciones y evitar basura de datos. Aquí entran tipos de activo, estados, áreas, ubicaciones, marcas, modelos opcionales, roles y tipos de licencia.

6. Relación entre módulos y por qué frontend los necesita

6.1 Autenticación y roles
Frontend necesita saber quién entró y qué puede ver o hacer. Sin esto no puede ocultar/mostrar acciones ni proteger secciones.

6.2 Inventario de activos
Es el corazón del sistema. Frontend necesita:
- listado paginado
- detalle
- crear
- editar
- cambiar estado
- buscar y filtrar

6.3 Asignaciones
Frontend necesita poder ver a quién está asignado un activo, reasignarlo y consultar historial. No basta con guardar una columna estática.

6.4 Licencias
Frontend necesita crear licencias, ver vencimientos, asociarlas a activo/usuario y listar asignaciones para paneles y detalle.

6.5 Auditoría
Frontend necesita consultas por usuario, fecha y acción para pantallas de trazabilidad o soporte.

6.6 Reportes
Frontend necesita endpoints listos para dashboard, tablas y exportación sin ponerse a recomputar lógica pesada en cliente.

6.7 Catálogos
Frontend necesita combos consistentes para tipo, estado, área, ubicación, rol y tipo de licencia. Eso no debe estar hardcodeado en React.

9. Reglas de negocio mínimas que backend debe respetar

9.1 Unicidad
- codigoInterno de activo debe ser único.
- serialNumber de activo debe ser único cuando exista.
- email de usuario debe ser único.
- codigoInterno de licencia idealmente único cuando exista.

9.2 Integridad referencial
No se debe permitir guardar referencias a tipo, estado, área, ubicación, usuario o licencia inexistentes.

9.3 Asignación vigente única
Un activo no debe tener dos asignaciones vigentes simultáneas si el modelo asume una sola responsabilidad actual.

9.4 Historial obligatorio
Cambios de estado y reasignaciones deben dejar historial.

9.5 Fechas válidas
- fechaFinVigencia no puede ser menor a fechaInicioVigencia.
- fechaGarantiaFin no puede ser menor a fechaAdquisicion cuando ambas existan.

9.6 Borrado controlado
No eliminar físicamente información que afecte trazabilidad salvo decisión explícita y protegida.

9.7 Roles
- admin: CRUD total, reportes, importación, auditoría, gestión de usuarios y catálogos.
- tecnico: CRUD operativo de activos, asignaciones, cambios de estado, licencias según política.
- consulta: solo lectura de módulos permitidos.

10. Validaciones que frontend espera que backend haga

Usuarios:

11. Contrato de respuesta recomendado para toda la API
Para evitar retrabajo frontend, uniformar respuestas.

Éxito:
{
  "success": true,
  "message": "Activo creado correctamente",
  "data": {...}
}

Éxito con listado:
{
  "success": true,
  "data": {
    "items": [...],
    "pagination": {
      "page": 1,
      "pageSize": 10,
      "totalItems": 100,
      "totalPages": 10
    }
  }
}

Error validación:
{
  "success": false,
  "message": "Error de validación",
  "errors": {
    "codigoInterno": ["El código interno es obligatorio"],
    "serialNumber": ["El serial ya existe"]
  }
}

Error negocio:
{
  "success": false,
  "message": "No se puede asignar la licencia porque el activo está inactivo"
}

15. Qué no debe hacer backend si quiere evitar retrabajo
- No devolver ids sin labels mínimos cuando frontend necesita mostrar tablas.
- No hardcodear catálogos solo en frontend.
- No mezclar estado operativo con borrado lógico sin claridad.
- No editar asignaciones históricas; deben cerrarse y abrirse nuevas.
- No perder payload anterior en cambios auditables.
- No exponer respuestas distintas por módulo sin estándar.
- No dejar importaciones sin trazabilidad de errores.
- No asumir que frontend resolverá integridad relacional.

17. Necesidades concretas del frontend para evitar fricción
Frontend necesita que backend entregue:
- ids y nombres legibles de relaciones en listados y detalles
- paginación consistente
- filtros estables
- mensajes de error claros
- catálogos para selects
- endpoints específicos para acciones, no todo metido en update general
- timestamps para mostrar historial
- respuestas predecibles
- posibilidad de pedir detalle expandido cuando una pantalla lo necesite

18. Supuestos funcionales que este documento toma para aterrizar el preliminar
Estas decisiones son inferencias sanas porque el preliminar no baja a detalle de campos:
- Un activo tiene una asignación actual y puede tener historial de asignaciones.
- Un activo tiene un estado actual y un historial de estados.
- Las licencias pueden asignarse a activo o usuario.
- Habrá catálogos para evitar texto libre incontrolado.
- El borrado será lógico en entidades principales.
- Habrá JWT o equivalente para autenticación.
- La auditoría se modelará como tabla propia.

Si backend decide algo distinto, debe documentarlo antes para no romper frontend.

19. Resumen ejecutivo para el backend
El backend de SIGAT-ES no debe construirse como un CRUD aislado de una sola tabla. El preliminar exige un sistema centralizado con trazabilidad, clasificación, estados, responsables, ubicaciones, licencias, reportes, autenticación, roles, migración inicial y evidencias de QA. Para soportar eso sin romper la base de datos y sin causar retrabajo a frontend, la API debe organizarse por módulos, sostenerse en una base relacional normalizada, manejar catálogos, guardar historial de cambios críticos y exponer endpoints pensados para las pantallas reales del sistema.

21. Nota final de alineación
Si este documento se adopta como fuente de verdad backend, cualquier cambio posterior en naming, payloads, relaciones o reglas debe versionarse y comunicarse antes de implementarse. El mayor riesgo de retrabajo aquí no es técnico, sino falta de contrato temprano entre frontend y backend.
